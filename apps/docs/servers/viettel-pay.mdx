## Overview

ViettelPay payment gateway integration (mock-only). Create payment requests, query transaction status, and process refunds. This server provides a REST+HMAC-SHA256 API approximation ‚ÄĒ the actual ViettelPay production API uses SOAP+RSA.

## Tools

| Tool | Description |
|------|-------------|
| `viettel_pay_create_payment` | Create a ViettelPay payment request. Returns a payment URL for the customer. Note: mock-only server. |
| `viettel_pay_query_status` | Check ViettelPay transaction status by `transactionId`. Note: mock-only server. |
| `viettel_pay_refund` | Refund a ViettelPay transaction. Note: mock-only server. |

---

## viettel_pay_create_payment

Create a ViettelPay payment request. Returns a payment URL and `transactionId`.

**Parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `amount` | number | Yes | Payment amount in VND (e.g., `50000`) |
| `orderInfo` | string | Yes | Human-readable order description |
| `orderId` | string | No | Unique order ID (auto-generated if omitted) |
| `returnUrl` | string | No | URL to redirect customer after payment |

**Example Response:**

```json
{
  "transactionId": "VTP1711234567890",
  "paymentUrl": "https://sandbox.viettelpay.vn/vtpay-api/pay?transId=VTP1711234567890",
  "status": "PENDING",
  "amount": 50000,
  "_mock": true
}
```

## Common Workflows

**1. Create and check a payment:**
1. Call `viettel_pay_create_payment` with `amount` and `orderInfo` ‚Üí receive `paymentUrl` and `transactionId`
2. Share the `paymentUrl` with the customer to complete payment
3. Call `viettel_pay_query_status` with the `transactionId` ‚Üí confirm `status: "SUCCESS"`

**2. Create and refund a payment:**
1. Call `viettel_pay_create_payment` ‚Üí receive `transactionId`
2. Call `viettel_pay_refund` with `transactionId` and refund amount

## Mock Deviations

This server models a REST+HMAC-SHA256 API. The real ViettelPay API uses SOAP+RSA. Key deviations:

- **Protocol:** REST JSON instead of SOAP XML
- **Authentication:** HMAC-SHA256 instead of RSA certificate signing
- **Endpoints:** Approximated REST paths ‚ÄĒ real SOAP WSDL differs significantly

All specific deviations are documented in `MOCK_DEVIATIONS.md` in the server repository.
